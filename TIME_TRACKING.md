# Time Tracking System: Интегрированный подход

## 1. Миссия и Философия

Этот документ описывает архитектуру и принципы работы **интегрированной** системы отслеживания времени. Основная цель — глубоко встроить механизм логирования времени в существующую систему отслеживания заказов, чтобы обеспечить максимальную целостность данных и простоту аналитики.

Мы жертвуем универсальностью ради надежности и скорости получения инсайтов в рамках нашего основного бизнес-процесса.

## 2. Архитектура: "3-5 Мультианкоров"

Ключевым архитектурным решением является использование **"мультианкорной"** таблицы `time_log`, которая напрямую ссылается на ключевые сущности нашей основной схемы данных (`product`, `order` и т.д.) через внешние ключи (`FOREIGN KEY`).

### 2.1. Схема таблиц

**Таблица `time_log_step` (Справочник шагов):**
*   **Назначение:** Хранит предопределенный список всех возможных шагов (действий), время выполнения которых мы хотим измерить. Это предотвращает ошибки и стандартизирует аналитику.
*   **Структура:**
    | Название колонки   | Тип данных | Описание                                  |
    |:-------------------|:-----------|:------------------------------------------|
    | `id`               | `INTEGER`  | Primary Key.                              |
    | `step_name`        | `TEXT`     | Уникальное кодовое имя (`product_research`).|
    | `step_title`       | `TEXT`     | Человекочитаемое название ("Исследование товара"). |
    | `step_description` | `TEXT`     | Подробное описание шага.                  |

**Таблица `time_log` (Журнал времени):**
*   **Назначение:** Основная таблица для хранения всех временных логов.
*   **Особенность:** Использует принцип **"разреженных колонок" (`sparse columns`)**. Большинство `fk_` полей в каждой строке будут `NULL`, и это является нормальным поведением для данной архитектуры.
*   **Структура:**
    | Название колонки  | Тип данных | Описание                                      |
    |:------------------|:-----------|:----------------------------------------------|
    | `id`              | `INTEGER`  | Primary Key.                                  |
    | `fk_step_id`      | `INTEGER`  | Внешний ключ на `time_log_step.id`.           |
    | `fk_product_id`   | `INTEGER`  | **Анкор 1:** Внешний ключ на `product.id`.    |
    | `fk_order_id`     | `INTEGER`  | **Анкор 2:** Внешний ключ на `order.id`.      |
    | `...`             | `INTEGER`  | *Здесь могут быть добавлены будущие анкоры (`fk_dispute_id` и т.д.)* |
    | `start_ts`        | `DATETIME` | Начало выполнения.                            |
    | `end_ts`          | `DATETIME` | Окончание выполнения.                         |
    | `duration_minutes`| `INTEGER`  | Длительность.                                 |
    | `notes`           | `TEXT`     | Заметки.                                      |

### 2.2. Принцип работы
1.  **Ручное логирование:** На начальном этапе данные вносятся вручную.
2.  **Целостность данных:** База данных сама следит за тем, чтобы каждая запись в `time_log` ссылалась на реально существующий шаг и реально существующую сущность (продукт, заказ).
3.  **Аналитика:** Для получения аналитических срезов используются простые и быстрые SQL-запросы с `LEFT JOIN`. Например, чтобы посчитать суммарное время, потраченное на все задачи, связанные с определенным продуктом, достаточно простого запроса по `fk_product_id`.

## 3. Плюсы и минусы

*   **Плюсы:**
    *   **Высокая надежность:** Гарантируется целостность данных на уровне БД.
    *   **Простота аналитики:** Легкие и быстрые запросы.
    *   **Низкий порог входа:** Легко реализовать в рамках существующей SQL-инфраструктуры.
*   **Минусы:**
    *   **Низкая гибкость:** Требует миграции схемы (`ALTER TABLE`) при добавлении нового типа отслеживаемой сущности.
    *   **Непереиспользуемость:** Решение тесно связано с конкретной схемой данных и не может быть легко перенесено в другой проект.
